## 1. 정보 구조 (Information Architecture)

### 1-1. 섹션 목록과 순서

| # | 섹션 ID | 섹션명 | 역할 |
|---|---------|--------|------|
| 1 | `#hero` | 히어로 | 방문자의 첫인상. 한 줄 정체성 + 핵심 메시지 |
| 2 | `#perspective` | 왜 언어가 아니라 방식인가 | 이 페이지의 관점을 설명하는 도입부. 방문자가 "기술 목록"을 기대할 때 인식을 전환시키는 역할 |

| 4 | `#workflow` | 작업 흐름 시각화 | 설계→구현→검토→배포 파이프라인을 시각적으로 제시. 역량이 실제 흐름 속 어느 단계에 적용되는지 맵핑 |

### 1-2. 섹션별 콘텐츠 상세

#### § 1 — 히어로 (`#hero`)
- **한 줄 정체성:** "AI와 협업하는 방식을 설계하는 사람"  
  *(직함·도구명 배제. '방식을 설계한다'는 행위 중심)*
- **핵심 메시지 (서브헤드):** "무엇을 쓰느냐가 아니라, 어떻게 다루느냐가 역량입니다."
- **CTA:** '역량 살펴보기' 버튼 → `#competencies` 앵커

#### § 2 — 관점 도입부 (`#perspective`)
- 방문자 질문 "어떤 언어 쓰세요?"를 정면으로 받아치는 구조
- **3단 구성:**
  1. 도발적 도입: "언어는 도구입니다. 도구는 교체됩니다."
  2. 핵심 주장: AI 협업에서 지속 가치를 갖는 것은 방법론
  3. 전환: "그래서 이 페이지는 도구 목록 대신 방식을 소개합니다."
- 분량: 3~4문장 (스캔 가능한 분량 유지)

#### § 3 — 역량 7가지 (`#competencies`)

| # | 역량명 | 1줄 정의 | 1줄 효과 |
|---|--------|----------|----------|
| 1 | 컨텍스트 엔지니어링 | AI가 올바른 판단을 내리도록 배경·제약·목표를 구조화된 언어로 설계한다 | 반복 수정 없이 첫 시도에서 원하는 결과를 얻는 효율을 확보한다 |
| 2 | 작업 분해 | 복잡한 목표를 AI가 처리 가능한 최소 단위로 나누고 순서를 설계한다 | 대형 프로젝트도 예측 가능한 단계로 진행되어 품질 관리가 가능해진다 |
| 3 | 역할 분업 (오케스트레이션) | 여러 AI 에이전트(또는 페르소나)에 역할을 배분하고 출력을 통합한다 | 단일 모델의 한계를 넘어 전문화된 시각들이 하나의 결과물로 수렴된다 |
| 4 | 검증·승인 게이트 | AI 출력물을 다음 단계로 넘기기 전 사람의 판단이 개입하는 체크포인트를 설계한다 | 오류가 누적되기 전에 조기 발견되어 전체 재작업 비용이 최소화된다 |
| 5 | 검증 루프 | AI가 스스로 자신의 출력을 기준에 따라 반복 검토·수정하는 프로세스를 설계한다 | 사람의 개입 없이도 일정 수준 이상의 품질이 자동으로 보장된다 |
| 6 | 의사결정 기록 | AI와의 협업 과정에서 내려진 판단 근거를 구조화된 형태로 추적·보존한다 | 이후 유사 문제 발생 시 동일한 사고 경로를 재현하거나 오류 원인을 신속히 파악한다 |
| 7 | 재사용 자산화 | 반복 사용 가능한 프롬프트·템플릿·컨텍스트 블록으로 협업 결과를 구조화한다 | 새 프로젝트 착수 시 초기 비용이 줄고 품질의 일관성이 높아진다 |

#### § 4 — 작업 흐름 시각화 (`#workflow`)
- **파이프라인 4단계:** `설계(Architect)` → `구현(Implementer)` → `검토(Reviewer)` → `배포(Deploy)`
- 각 단계 아래에 위 7가지 역량 중 해당 단계에서 주로 작동하는 역량 번호 표기
  - 설계: ①컨텍스트 엔지니어링, ②작업 분해
  - 구현: ③역할 분업, ⑦재사용 자산화
  - 검토: ④검증·승인 게이트, ⑤검증 루프
  - 배포: ⑥의사결정 기록
- 구현 방식: CSS Flexbox 화살표 다이어그램 (SVG 또는 순수 CSS)

#### § 5 — 사례 (`#cases`) — 플레이스홀더
```

```

## 2. 톤 & 보이스

### 제안 형용사 3개

| 형용사 | 근거 |
|--------|------|
| **명료한 (Lucid)** | 방법론을 설명하는 페이지는 추상적 어휘가 많아지기 쉽다. 각 문장이 한 번에 이해되지 않으면 역량 자체의 신뢰도가 떨어진다. |
| **자신감 있는 (Assured)** | "~할 수 있습니다"가 아닌 "~합니다" 형식. 역량을 소개하는 글이 주저하면 설득력이 없다. |
| **실용적인 (Pragmatic)** | 이상론이 아닌 실제 효과(Effect)를 중심으로 기술. 방문자가 "그래서 뭐가 좋은가"를 즉시 알 수 있어야 한다. |

### 피해야 할 어조
- 겸손 과잉: "미흡하지만", "부족하지만" — 방문자의 신뢰를 스스로 깎는다
- 기술 나열식: "Python, Docker, AWS …" 형태의 목록 — 핵심 원칙 위반
- 추상 과잉: "패러다임", "시너지" 등 의미 없이 무게만 있는 단어

## 3. 보편성 점검 체크리스트

콘텐츠 작성 전후로 아래를 점검한다.

### A. 역량 소개 문장 점검
- [ ] 이 문장에 특정 프로그래밍 언어명이 등장하는가?
- [ ] 이 문장에 특정 프레임워크/라이브러리명이 등장하는가?
- [ ] 이 문장에 특정 데이터베이스 제품명이 등장하는가?
- [ ] 이 문장에 특정 클라우드/SaaS 서비스명이 등장하는가?
- [ ] 위 항목 중 하나라도 해당되면 → **방법론 관점으로 재작성**

### B. 사례 설명 문장 점검 (STEP 6 시)
- [ ] 사례에서 도구명이 등장한다면, 그 문장의 주어는 도구인가 사람인가?
  - 도구가 주어: ❌ "Python으로 분석했다" → ✅ "데이터 탐색 과정에서 스크립팅 도구를 활용했다"
- [ ] 도구명 언급이 **결과를 설명하는 데 꼭 필요한가?** 불필요하면 삭제.
- [ ] 역량 태그가 붙은 카드의 태그는 7가지 역량 목록에만 있는가?

### C. 전체 페이지 최종 점검
- [ ] 히어로, 도입부, 역량 섹션 어디에도 특정 기술이 "역량"으로 등장하지 않는가?
- [ ] "할 수 있는 기술: …" 형태의 목록이 없는가?
- [ ] 역량 7가지 각각에 **정의(What)** 와 **효과(Why it matters)** 가 모두 있는가?

## 4. 디자인 방향

### 4-1. 색상

| 역할 | 값 (제안) | 의도 |
|------|-----------|------|
| 배경 | `#0F1117` (거의 검정) | 콘텐츠에 집중, 고급스러운 인상 |
| 표면(카드) | `#1A1D27` | 배경과 구분되는 레이어감 |
| 주요 텍스트 | `#E8EAF0` | 다크 배경에서 가독성 확보 |
| 보조 텍스트 | `#8B90A0` | 정의·부연 설명 등 2차 정보 |
| 액센트 | `#4F7EFF` (쿨한 파랑) | CTA 버튼, 역량 번호, 링크 hover |
| 구분선 | `#2A2D3A` | 섹션 간 경계 최소화 |

> 대안: 화이트 배경(`#FAFAFA`) + 다크 텍스트(`#1A1A2E`) + 액센트 `#2F54EB` — 밝은 환경 선호 시 사용자 선택

### 4-2. 타이포그래피

| 용도 | 폰트 | 크기 (데스크탑) |
|------|------|----------------|
| 히어로 헤드라인 | 시스템 폰트 스택 (`-apple-system, 'Segoe UI', sans-serif`) | `clamp(2.5rem, 5vw, 4rem)` |
| 섹션 제목 | 동일 | `clamp(1.5rem, 3vw, 2rem)` |
| 역량명 | 동일, `font-weight: 700` | `1.125rem` |
| 본문/정의 | 동일, `font-weight: 400` | `1rem`, `line-height: 1.7` |
| 효과 문장 | 동일, 이탤릭 또는 액센트 색 | `0.9rem` |

> 외부 웹폰트 미사용 — 시스템 폰트로 로딩 비용 제로

### 4-3. 레이아웃

- **단일 열 중심, 최대 너비 `800px`** — 읽기 최적화
- 역량 7가지: `grid-template-columns: repeat(auto-fill, minmax(300px, 1fr))` — 모바일에서 자동 1열 전환
- 워크플로우: Flexbox 수평 배열 → 모바일에서 수직 스택
- 섹션 간격: `padding: 5rem 1.5rem` — 여백으로 구조를 표현
- 애니메이션: `IntersectionObserver` + CSS `@keyframes fadeInUp` — JS 의존 최소화, 접근성 `prefers-reduced-motion` 대응

### 4-4. 구현 범위 제약
- 빌드 도구 없음: 순수 `index.html` + `style.css` + `main.js` (선택)
- 외부 CDN: 아이콘만 허용 (예: Feather Icons SVG 스프라이트 — 인라인 삽입 가능)
- 이미지: 없거나 최소화 (SVG 아이콘만)
- 총 파일: 3개 이하
