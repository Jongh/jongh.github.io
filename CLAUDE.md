# PROJECT: AI 기술 스택 포트폴리오 페이지

## 목적
AI 활용 방법론을 중심으로 자기소개·업무 이력·핵심 사례·AI 특장점을 소개하는 개인 포트폴리오 페이지.  
GitHub Pages 정적 배포 (빌드 단계 없음, 프레임워크 비종속).  
콘텐츠 언어: 한국어  
배포 URL: https://jongh.github.io

## 핵심 원칙 (절대 위반 금지)
1. 특정 프로그래밍 언어/프레임워크/DB/제품명을 "역량"으로 소개하지 않는다.
   사례 설명에서 불가피하게 언급할 때도 '기술 자체'가 아닌 '어떻게 다뤘는가'라는 방법론 관점으로 추상화한다.
2. 소개의 주제는 항상 'AI를 다루는 방식'이다.
3. 코드보다 구조와 콘텐츠 설계가 먼저다. 불명확하면 구현하지 말고 질문한다.
4. 단계마다 산출물을 보고하고, 사용자 승인 없이 다음 단계로 넘어가거나 기존 파일을 임의로 덮어쓰지 않는다.

## AI 활용 특장점 (기존 프로젝트 구조에서 도출)

> 이 목록은 CLAUDE.md와 프로젝트 자체 구조를 🔍 Reviewer 관점으로 분석한 결과다.
> 포트폴리오 `#ai-strengths` 섹션의 실증 근거로 활용한다.

| # | 특장점 | 근거 (이 프로젝트에서의 실증) |
|---|--------|-------------------------------|
| 1 | **컨텍스트 엔지니어링** | CLAUDE.md 자체가 산출물 — 배경·원칙·상태·결정을 구조화된 단일 문서로 관리하여 세션 간 맥락 유실 없음 |
| 2 | **작업 분해** | STEP 1→6의 단계별 진행 상태 테이블로 대형 프로젝트를 예측 가능한 최소 단위로 분해 |
| 3 | **역할 분업 (오케스트레이션)** | Architect·Writer·Implementer·Reviewer 4개 페르소나를 명확히 분리하여 역할 경계 충돌 없이 전문화된 출력 통합 |
| 4 | **검증·승인 게이트** | "사용자 승인 없이 다음 단계로 넘어가지 않는다"를 CLAUDE.md 규칙에 박아 넣음 — 오류 누적 전 조기 발견 |
| 5 | **검증 루프** | 보편성 체크리스트(A/B/C)를 사전 정의하여 Writer와 Reviewer가 자동 자기검증 — 반복 수정 비용 최소화 |
| 6 | **의사결정 기록** | '주요 결정 사항' 테이블로 모든 판단 근거를 즉각 문서화 — 향후 세션에서 동일 논의 반복 불필요 |
| 7 | **재사용 자산화** | `docs/ia-spec.md`(Architect) · `docs/copy-draft.md`(Writer)를 역할별 분리 보관 → Single Source of Truth |

## 소개할 보편적 AI 협업 역량 7가지
1. 컨텍스트 엔지니어링
2. 작업 분해
3. 역할 분업 (오케스트레이션)
4. 검증·승인 게이트
5. 검증 루프
6. 의사결정 기록
7. 재사용 자산화

## 작업 페르소나
- 🏛️ Architect  — 구조/콘텐츠 설계 (코드 없음)
- ✍️ Writer     — 카피 작성
- 🔨 Implementer — 정적 사이트 구현
- 🔍 Reviewer   — 비판적 검토 (보편성·접근성·성능 중심)

## 작업 규칙
- 각 단계 완료 시 산출물 보고 → 사용자 승인 후 다음 단계 진행
- 기존 파일 덮어쓰기 전 반드시 확인
- 역량 소개는 항상 방법론 관점 유지
